# 项目验收标准

本文档定义了 AI 速录助手（安卓版）项目的验收标准，所有交付物必须满足以下要求才能通过验收。

---

## 1 硬性门槛（一票否决）

核心原则：代码必须能跑，且必须跑的是题目要求的内容。

### 1.1 绝对的可运行性

| 要求 | 说明 |
|------|------|
| 一键构建 | 项目必须支持 `./gradlew assembleDebug` 构建，无报错 |
| 一键安装 | APK 必须能正常安装到 Android 8.0+ 设备 |
| 环境隔离 | 不得依赖开发者本地的特定路径、全局环境变量或未声明的系统库 |
| 文档一致 | README 中的安装和运行步骤必须真实有效 |

### 1.2 严格的切题性

| 要求 | 说明 |
|------|------|
| 核心功能完整 | 必须实现悬浮球、按住录音、语音转文字、AI润色、自动输入五大核心功能 |
| 禁止擅自简化 | 严禁通过大幅削减功能来降低开发难度 |
| 交互符合规范 | 按住悬浮球录音、松开结束、文字自动上屏的交互流程必须完整实现 |

---

## 2 交付完整性

核心原则：交付的是产品雏形，而不是代码片段。

### 2.1 具备工程化结构

| 要求 | 说明 |
|------|------|
| 项目形态 | 必须是完整的 Android 工程项目，具备清晰的目录结构 |
| 模块分层 | UI层、Service层、Repository层、Model层职责分离 |
| 配置完整 | 必须包含完整的 build.gradle.kts、AndroidManifest.xml、proguard-rules.pro |

### 2.2 真实逻辑实现

| 要求 | 说明 |
|------|------|
| 逻辑真实 | ASR识别和AI润色必须调用真实API（可使用免费额度） |
| 禁止硬编码 | 不能直接返回写死的文本，必须包含真实的API调用逻辑 |
| 接口可配置 | API Key 通过 BuildConfig 注入，不硬编码在代码中 |

---

## 3 工程与架构质量

核心原则：代码需具备可维护性，符合行业通用规范。

### 3.1 架构分层合理

| 要求 | 说明 |
|------|------|
| MVVM架构 | ViewModel、Repository、Model职责分离 |
| 单文件限制 | 单个文件不超过 500 行，单函数不超过 50 行 |
| 命名语义化 | 目录命名具有语义（如 ui/service/repository/util） |

### 3.2 代码整洁度

| 要求 | 说明 |
|------|------|
| 清理冗余 | 删除被注释的废弃代码、调试用的 print/log 语句 |
| 配置脱敏 | 确保配置文件不包含个人密钥、内网 IP |
| 无垃圾文件 | 不提交 build/、.gradle/、.idea/ 等构建产物 |

### 3.3 可维护性

| 要求 | 说明 |
|------|------|
| 拒绝魔术数字 | 使用常量定义，避免硬编码数字 |
| 避免深层嵌套 | if-else 嵌套不超过 3 层 |
| 注释必要 | 复杂逻辑必须有中文注释说明 |

---

## 4 测试标准

### 4.1 单元测试要求

| 要求 | 说明 |
|------|------|
| 测试目录 | `src/test/` 目录下存放单元测试 |
| 测试框架 | 使用 JUnit 5 + MockK |
| 覆盖范围 | ViewModel、Repository、工具类必须有测试 |
| 覆盖率 | 核心业务逻辑覆盖率 ≥ 80% |

### 4.2 仪器测试要求

| 要求 | 说明 |
|------|------|
| 测试目录 | `src/androidTest/` 目录下存放仪器测试 |
| 测试内容 | Room 数据库 CRUD 操作必须有测试 |
| 执行脚本 | 提供 `run_tests.sh` 一键执行脚本 |

### 4.3 测试输出要求

| 要求 | 说明 |
|------|------|
| 结果清晰 | 测试执行后输出成功/失败状态 |
| 失败详情 | 失败时输出错误信息和堆栈 |
| 结果汇总 | 输出测试总数、通过数、失败数 |

---

## 5 工程细节与专业度

核心原则：按生产级代码的标准要求自己。

### 5.1 健壮的错误处理

| 要求 | 说明 |
|------|------|
| 错误提示 | API 失败时显示友好的错误提示，不直接显示堆栈 |
| 降级处理 | 网络失败时提供重试选项 |
| 异常捕获 | 使用 sealed class 封装错误类型 |

### 5.2 规范的日志记录

| 要求 | 说明 |
|------|------|
| 有效日志 | 关键业务流程（录音、识别、润色）必须有日志 |
| 日志质量 | 日志包含必要上下文，不输出无意义内容 |
| 脱敏处理 | API Key 等敏感信息在日志中脱敏 |

### 5.3 安全与参数校验

| 要求 | 说明 |
|------|------|
| 输入防御 | 对用户输入进行合法性校验 |
| 权限检查 | 启动时检查必要权限，未授予时引导用户 |
| API安全 | API Key 通过 BuildConfig 注入，不硬编码 |

---

## 6 需求理解深度

核心原则：做完、做对、做好。

### 6.1 识别隐含约束

| 要求 | 说明 |
|------|------|
| 业务闭环 | 录音→识别→润色→输入的完整流程必须闭环 |
| 权限引导 | 悬浮窗、麦克风、辅助功能权限必须有引导页面 |
| 后台保活 | 服务在后台不被系统杀死 |

### 6.2 拒绝机械式翻译

| 要求 | 说明 |
|------|------|
| 场景适配 | 适配安卓特有的交互方式（按住录音、悬浮球） |
| 系统集成 | 正确使用 AccessibilityService 实现文字注入 |

---

## 7 美观度（移动端题目）

核心原则：界面应整洁、现代，具备基本的交互可用性。

### 7.1 视觉规范

| 要求 | 说明 |
|------|------|
| 布局整齐 | 元素对齐、间距统一，无内容溢出 |
| 配色和谐 | 主色调统一，对比度适宜 |
| 现代化 | 使用 Material Design 3 组件 |

### 7.2 交互体验

| 要求 | 说明 |
|------|------|
| 操作反馈 | 悬浮球按住时有震动反馈，录音中有波形显示 |
| 流程顺畅 | 权限引导→悬浮球显示→录音→输入的流程顺畅 |
| 状态可见 | 录音中、识别中、润色中有状态提示 |

## 8 不可验收情况

以下情况视为交付物不可用，直接判定为不合格：

### 8.1 构建失败

| 情况 | 说明 |
|------|------|
| 构建报错 | `./gradlew assembleDebug` 构建过程报错 |
| 安装失败 | APK 无法安装到标准 Android 设备 |
| 运行崩溃 | 应用启动后立即崩溃或无法保持运行 |

### 8.2 依赖人工干预

| 情况 | 说明 |
|------|------|
| 手动配置 | 需要手动创建文件或导入配置才能运行 |
| 交互式输入 | 启动时需要用户手动输入配置 |
| 口头沟通 | README 未写明，需要额外沟通才能运行 |

### 8.3 环境隔离失效

| 情况 | 说明 |
|------|------|
| 路径依赖 | 代码包含开发者本地绝对路径 |
| 全局依赖 | 依赖未在 build.gradle 中声明的全局库 |
| 私有资源 | 引用私有镜像仓库或无法访问的资源 |

### 8.4 文档与实际行为不一致

| 情况 | 说明 |
|------|------|
| 虚假文档 | README 中的步骤与实际代码行为不符 |
| 功能缺失 | 文档声称的功能实际未实现 |
| 验证不可达 | 按文档操作无法验证核心功能 |

### 8.5 依赖污染

| 情况 | 说明 |
|------|------|
| 构建产物 | 项目包含 build/、.gradle/ 等构建产物 |
| IDE配置 | 包含 .idea/ 等 IDE 配置文件 |
| 依赖目录 | 包含 .cxx/、.externalNativeBuild/ 等缓存目录 |

## 10 验收检查清单

验收时按以下清单逐项检查：

### 10.1 构建检查

- [ ] `./gradlew assembleDebug` 构建成功
- [ ] APK 文件生成正常
- [ ] APK 可安装到 Android 8.0+ 设备

### 10.2 功能检查

- [ ] 悬浮球在任意界面显示
- [ ] 悬浮球可拖动、自动贴边
- [ ] 按住悬浮球开始录音（有震动反馈）
- [ ] 松开悬浮球结束录音
- [ ] 录音中有音量波形显示
- [ ] 语音识别返回正确文本
- [ ] AI润色输出符合选择的风格
- [ ] 文字自动输入到其他App的输入框
- [ ] 历史记录页面正常显示
- [ ] 设置页面功能正常

### 10.3 权限检查

- [ ] 悬浮窗权限引导正常
- [ ] 麦克风权限申请正常
- [ ] 辅助功能权限引导正常
- [ ] 权限被拒绝时有友好提示

### 10.4 代码检查

- [ ] 目录结构清晰
- [ ] MVVM架构分层合理
- [ ] 无硬编码API Key
- [ ] 无废弃代码和调试日志
- [ ] 无构建产物提交

### 10.5 测试检查

- [ ] 单元测试可执行
- [ ] 仪器测试可执行
- [ ] 测试覆盖率 ≥ 80%
- [ ] 测试结果输出清晰

### 10.6 文档检查

- [ ] README 安装步骤正确
- [ ] README 功能说明完整
- [ ] README 包含测试验证步骤
