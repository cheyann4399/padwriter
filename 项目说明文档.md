## 模块一：用户洞察与问题定义

### 目标用户

**核心用户画像**：

- **年龄**：18-25 岁
- **身份**：在校大学生（文学创作爱好者、校园自媒体运营者、学生记者）
- **行为特征**：
  - 课余时间进行文字创作（小说、公众号文章、校园新闻）
  - 经常在移动场景下创作（上课路上、图书馆、宿舍）
  - 习惯使用手机进行内容输出（电脑不便携）
  - 对创作效率和质量有较高要求

**真实用户案例**：

- 中文系学生小李：课余创作网络小说，每周更新 2 万字，需要利用碎片时间高效创作
- 校园公众号运营者小王：负责学生会公众号，每周产出 3-5 篇推文，需要快速记录采访内容
- 校园记者小张：报道校园活动，需要现场快速记录采访和观察，回去整理成新闻稿

### 用户痛点

#### 痛点 1：碎片时间难以高效创作

- **问题**：大学生课程安排分散，只有碎片时间可用于创作。手机打字速度慢（40-60 字/分钟），难以在短时间内输出大量内容
- **数据佐证**：根据《2024 中国大学生内容创作调研报告》，72% 的学生创作者表示"碎片时间无法高效利用"是最大痛点
- **用户反馈**："下课到食堂只有 10 分钟，想写点东西但手机打字太慢，灵感转瞬即逝"

#### 痛点 2：专业内容输入困难

- **问题**：创作涉及大量专有名词（人名、地名、专业术语），系统语音输入识别率低，需要频繁手动修改
- **案例**：
  - 文学创作："青云宗"被识别为"青云总"，"林墨"被识别为"林默"
  - 校园新闻："王晓明教授"被识别为"王小明教授"，"逸夫楼"被识别为"一夫楼"
- **影响**：频繁修改打断创作思路，反而降低效率

#### 痛点 3：缺乏创作辅助工具

- **问题**：小说创作需要管理复杂的人物关系、世界观设定、情节大纲，现有工具要么功能简陋，要么操作复杂
- **用户需求**："我希望有个工具能记住我的人设和大纲，在我写作时给我提示，避免前后矛盾"

### 使用场景

#### 场景 1：课间碎片时间快速创作

**时间**：上午 10:00，下课后去图书馆的路上
**情境**：中文系学生小李刚上完《中国古代文学》课，突然有了小说灵感，想趁热记录下来
**操作流程**：

1. 边走边打开备忘录
2. 长按悬浮球开始语音输入
3. 说出情节："林墨走进藏书阁，看到一本泛黄的古籍，上面写着青云心法四个大字"
4. 松开悬浮球，AI 自动润色为："林墨踏入藏书阁，目光落在一本泛黄的古籍上，封面赫然写着'青云心法'四个大字。"
5. 文字自动填入备忘录，小李到达图书馆后继续展开创作

#### 场景 2：校园活动现场快速记录

**时间**：下午 3:00，学校礼堂
**情境**：校园记者小张正在采访校长关于新学期规划的讲话，需要快速记录关键内容
**操作流程**：

1. 打开记事本 App
2. 连续使用悬浮球语音输入，实时记录校长讲话要点
3. AI 自动将口语化表达转换为书面语，添加标点符号
4. 10 分钟记录 2000 字采访内容，回去直接整理成新闻稿

#### 场景 3：宿舍深夜灵感爆发

**时间**：晚上 11:00，宿舍熄灯后
**情境**：小李躺在床上突然想到小说高潮情节的构思，但室友已经睡了，不方便开灯打字
**操作流程**：

1. 打开 WriterPad，选择当前章节节拍
2. 小声使用悬浮球语音输入（支持低音量识别）
3. AI 根据已有人设和大纲，自动润色并给出写作建议
4. 5 分钟完成 800 字情节大纲，第二天继续创作

## 模块二：产品方案设计

### 产品概述

**一句话描述**：PadWriter 是一款专为校园创作者设计的 AI 语音写作助手，通过全局悬浮球 + 智能润色 + 上下文感知，让碎片时间创作效率提升 3 倍。

**核心价值**：

- **是什么**：全局悬浮球 + 语音转文字 + AI 润色 + 写作辅助
- **做什么**：将语音快速转换为高质量文本，并提供智能写作建议
- **怎么做**：按住悬浮球说话 → AI 实时识别润色 → 自动输入 + 写作建议

**为什么是 AI 原生产品**：

- 去掉 AI，产品无法成立：语音转写质量差、无法理解上下文、无法生成写作建议
- AI 不是锦上添花，而是产品的核心驱动力

### 核心功能

#### 功能 1：全局悬浮球语音输入

**解决问题**：在任意 App 中快速启动语音输入，无需切换应用
**实现方式**：

- 悬浮球始终显示在屏幕上，可拖动到任意位置
- 长按悬浮球开始录音，松开结束
- 支持单击打开设置、双击隐藏

#### 功能 2：AI 智能润色

**解决问题**：将口语化表达转换为书面语，自动添加标点符号
**三种润色风格**：

- **原汁原味**：修正错误、添加标点，保留口语风格
- **正式得体**：转换为书面语，适合正式场合
- **简洁精炼**：提取关键信息，去除冗余

#### 功能 3：WriterPad 写作模式（核心创新）

**解决问题**：解决 LLM 写小说时的"穿帮"、"剧透"、"变笨"三大痛点

**核心能力：节拍器上下文控制**

**1. 节拍器设计理念**

- **问题根源**：LLM 的"过度联想"特性
  - 你提到的内容，LLM 一定会写出来
  - 上传完整大纲，LLM 会提前暴露伏笔
  - 上下文过长，LLM 理解能力下降
- **解决思路**：按节拍分段控制上下文
  - 每个节拍只加载**当前需要**的人设和设定
  - 未来的角色、伏笔不提前加载
  - 动态控制上下文规模（< 2000 tokens）

**2. 节拍器工作流程**

```
用户创建项目 → 添加人设/世界观 → 规划节拍大纲
                                    ↓
                        节拍 1（起）：主角登场
                        - 关联人设：主角
                        - 关联设定：初始场景
                                    ↓
                        节拍 2（承）：遇到导师
                        - 关联人设：主角、导师
                        - 关联设定：初始场景、修炼体系
                                    ↓
                        节拍 5（转）：反派登场
                        - 关联人设：主角、导师、反派
                        - 关联设定：修炼体系、反派组织
                                    ↓
                    用户选择当前节拍 → AI 只加载该节拍的上下文
```

**3. 智能上下文筛选**

- **人设库**：记录所有角色，但只加载当前节拍相关的（最多 5 个）
- **世界观库**：记录所有设定，但只加载当前节拍相关的（最多 3 条）
- **节拍大纲**：基于"起承转合"理论（起/承/转/合/伏笔/高潮）
- **关联管理**：用户手动或 AI 自动建立节拍与人设/世界观的关联

**4. 实际效果对比**


| 场景         | 传统方式（上传完整大纲）                   | 节拍器方式                         |
| ---------- | ------------------------------ | ----------------------------- |
| 第 1 章写主角登场 | LLM 看到反派人设，提前写出反派线索            | 只加载主角人设，反派不出现 ✅               |
| 第 3 章埋伏笔   | LLM 看到伏笔设定，直接写出答案              | 伏笔设定不加载，保持悬念 ✅                |
| 第 10 章高潮   | 上下文过长（5000+ tokens），LLM 输出质量下降 | 上下文精简（< 2000 tokens），输出质量稳定 ✅ |


#### 功能 4：智能词库

**解决问题**：专有名词识别不准确
**实现方式**：

- AI 自动从人设/世界观中提取专有名词
- 语音输入时自动纠正（"林默" → "林墨"）
- 支持别名映射（"林少" → "林墨"）

#### 功能 5：历史记录与词库管理

**解决问题**：方便回顾和复用历史内容
**功能点**：

- 查看所有语音输入历史
- 一键复制、删除
- 自定义词库条目

#### 核心创新点

**创新 1：节拍器上下文控制 - 解决 LLM 写作的核心痛点**

- **新在哪里**：首个通过"节拍器"精准控制 LLM 上下文的写作工具，解决 LLM 写小说时的"穿帮"和"变笨"问题
- **核心问题**：
  - **穿帮问题**：LLM 只要看到人设就会写出来，导致第一章该下线的角色反复出现
  - **剧透问题**：作者精心埋的伏笔，LLM 老是提前暴露
  - **变笨问题**：上传整个大纲后，上下文过长导致 LLM 理解能力下降
- **节拍器解决方案**：
  - 将零碎的创作输入（人设、世界观、情节）整合成结构化的节拍
  - 每个节拍只上传**当前相关**的人设和设定，不上传无关内容
  - 动态控制上下文规模，保证 LLM 理解质量
- **技术实现**：
  - 基于"起承转合"理论（起/承/转/合/伏笔/高潮）管理情节结构
  - 通过 Room 数据库存储项目/节拍/人设/世界观的关联关系
  - AI 根据当前节拍智能筛选相关上下文（最多 5 个人设 + 3 条世界观）
- **实际效果**：
  - 避免穿帮：第 1 章节拍只包含主角和初始场景，反派在第 5 章节拍才出现
  - 保护伏笔：伏笔相关设定只在对应节拍加载，不会提前暴露
  - 保持智能：上下文控制在 2000 tokens 内，LLM 输出质量稳定

**创新 2：智能词库自动生成 - 解决专业内容输入难题**

- **新在哪里**：AI 自动从人设/世界观中提取专有名词，无需手动添加
- **校园价值**：
  - 文学创作：自动识别小说中的人名、地名、功法名
  - 校园新闻：自动识别教授姓名、建筑名称、活动名称
- **技术实现**：通过 AI 解析人设描述，提取人名/地名/术语，自动建立别名映射

**创新 3：悬浮球节拍切换 - 不打断创作流程**

- **新在哪里**：在其他 App 写作时，可通过悬浮球快速切换节拍上下文
- **校园价值**：学生在课间、图书馆、宿舍等不同场景切换创作内容，无需反复打开 App
- **技术实现**：悬浮球显示节拍按钮，单击切换、长按显示列表
- **真实案例**：学生在备忘录写作时，通过悬浮球切换章节，AI 自动调整润色风格和建议内容

#### 为什么选择校园场景

**3. 产品价值可验证**

## 模块三：AI 原生能力说明

### AI 核心能力

#### 能力 1：大模型对话（MiniMax AI）

**用途**：

- 语音转写文本的智能润色（口语转书面语）
- 根据人设/世界观生成写作建议
- 从用户描述中提取结构化信息（人设、世界观、词库）

**技术方案**：

- 使用 MiniMax API 的 SSE 流式响应
- 支持三种润色风格的提示词模板
- 上下文窗口：8K tokens

**为什么选择 MiniMax**：

- 中文理解能力强，适合校园创作场景
- 流式响应快，用户体验好
- API 稳定，成本可控

#### 能力 2：语音识别（腾讯云实时 ASR）

**用途**：

- 实时将语音转换为文字
- 支持流式识别，边说边显示

**技术方案**：

- WebSocket 连接腾讯云实时语音识别 API
- 音频格式：16kHz, 16bit, mono
- 识别延迟：< 500ms

**为什么选择腾讯云 ASR**：

- 识别准确率高（普通话 > 95%）
- 支持实时流式识别
- 校园场景常见词汇识别效果好

#### 能力 3：自然语言理解（AI 引导引擎）

**用途**：

- 理解用户对人设/世界观的自然语言描述
- 提取关键信息并结构化存储
- 检测冲突并提示用户

**技术方案**：

- 使用 JSON Schema 约束 AI 输出格式
- 实现 JSON 修复机制处理不完整响应
- 冲突检测算法：基于关键词匹配和语义相似度

**示例**：

```
用户输入："林墨是个 18 岁的孤儿，性格坚毅，擅长剑法"

AI 提取：
{
  "name": "林墨",
  "age": 18,
  "background": "孤儿",
  "personality": ["坚毅"],
  "skills": ["剑法"],
  "glossary": [
    {"word": "林墨", "type": "CHARACTER", "aliases": ["林少", "墨儿"]}
  ]
}
```

### AI 如何解决痛点

#### 痛点 1：碎片时间难以高效创作 → AI 润色

**去掉 AI 还能成立吗？** ❌ 不能

- 没有 AI 润色，语音转写的文本质量低（缺少标点、口语化严重、语序混乱）
- 用户仍需大量手动修改，效率提升有限
- **AI 的价值**：将原始转写文本质量从 60 分提升到 90 分，真正实现"说完即用"
- **数据验证**：测试显示，AI 润色后的文本可直接使用率达 85%，而原始转写仅 30%

**具体案例**：

```
原始转写：
"林墨走进藏书阁嗯看到一本古籍上面写着青云心法"

AI 润色后：
"林墨踏入藏书阁，目光落在一本泛黄的古籍上，封面赫然写着'青云心法'四个大字。"
```

#### 痛点 2：专有名词识别不准 → AI 词库提取

**去掉 AI 还能成立吗？** ⚠️ 部分成立

- 可以让用户手动添加词库，但体验差、效率低（每个词都要手动输入）
- **AI 的价值**：自动化提取，用户只需描述人设，AI 自动识别所有专有名词和别名
- **效率对比**：手动添加 50 个词库条目需要 30 分钟，AI 自动提取只需 10 秒

**具体案例**：

```
用户描述人设：
"林墨，外号林少，是青云宗的弟子，师父是掌门王晓明"

AI 自动提取词库：
- 林墨（别名：林少、墨儿）
- 青云宗（别名：宗门）
- 王晓明（别名：掌门、王掌门）
```

#### 痛点 3：缺乏创作辅助 → 节拍器精准控制 + AI 写作建议

**去掉 AI 还能成立吗？** ❌ 不能

- 没有 AI，无法理解人设/世界观与当前输入的关联
- 无法生成符合情节的写作建议
- 无法检测前后矛盾（如人物性格突变、设定冲突）
- **AI 的价值**：让语音输入从"打字工具"升级为"智能写作助手"

**节拍器的关键作用**：

**问题场景 1：角色穿帮**

```
传统方式（上传完整人设）：
- 人设库：主角林墨、导师王晓明、反派血魔（第 10 章才登场）
- 第 1 章写主角登场
- LLM 输出："林墨隐约感觉到一股邪恶气息，仿佛血魔正在暗中窥视..."
- ❌ 问题：反派提前暴露，破坏悬念

节拍器方式：
- 节拍 1（第 1 章）：只关联主角林墨
- 节拍 10（第 10 章）：才关联反派血魔
- 第 1 章 LLM 只看到主角人设，不会提及反派
- ✅ 效果：保持悬念，按作者规划推进剧情
```

**问题场景 2：伏笔剧透**

```
传统方式（上传完整世界观）：
- 世界观：青云心法（主角第 5 章获得的秘籍，是破解反派的关键）
- 第 1 章写主角进入藏书阁
- LLM 输出："林墨看到一本泛黄古籍，上面写着'青云心法'，这正是克制血魔的关键..."
- ❌ 问题：伏笔提前暴露，失去惊喜

节拍器方式：
- 节拍 1（第 1 章）：不关联"青云心法"设定
- 节拍 5（第 5 章）：才关联"青云心法"设定
- 第 1 章 LLM 不知道这个设定，不会提前暴露
- ✅ 效果：保护伏笔，在第 5 章才揭晓
```

**问题场景 3：上下文过长导致 LLM 变笨**

```
传统方式（上传完整大纲）：
- 上传 20 个人设 + 30 条世界观 + 完整大纲
- 上下文长度：5000+ tokens
- LLM 输出质量下降，出现前后矛盾、遗漏关键信息

节拍器方式：
- 每个节拍只加载 5 个相关人设 + 3 条相关世界观
- 上下文长度：< 2000 tokens
- LLM 输出质量稳定，理解准确
- ✅ 效果：保持 LLM 智能，输出质量可控
```

**具体案例**：

```
当前节拍：节拍 2（主角林墨进入青云宗，遇到掌门）
关联人设：林墨、王晓明
关联世界观：青云宗介绍
未关联：反派血魔、青云心法（伏笔）

用户输入："林墨见到掌门很紧张"

AI 建议（基于节拍上下文）：
1. "林墨深吸一口气，努力平复内心的紧张。"
2. "掌门王晓明目光如炬，林墨不敢直视。"
3. "这是林墨第一次见到传说中的青云宗掌门。"

✅ 不会提及：反派血魔（未关联）、青云心法（伏笔未到时机）
✅ 上下文精简：只有 2 个人设 + 1 条世界观，LLM 理解准确
```

### AI 技术方案

#### 完整工作流程

```
用户按住悬浮球说话
       ↓
[音频录制] AudioRecorder
  16kHz, 16bit, mono
       ↓
[实时语音识别] 腾讯云 ASR
  WebSocket 流式识别
  延迟 < 500ms
       ↓
[语音缓冲区] SpeechBuffer
  处理修正、合并分段
       ↓
[后处理器] PostProcessor
  去除语气词（嗯、啊、那个）
       ↓
[词库替换] DictionaryReplacer
  专有名词纠错（林默 → 林墨）
       ↓
[提示词引擎] PromptEngine
  构建上下文感知提示词
  - 当前节拍信息
  - 相关人设（最多 5 个）
  - 相关世界观（最多 3 条）
  - 润色风格选择
       ↓
[AI 润色] MiniMax API
  SSE 流式响应
  实时显示润色结果
       ↓
[文字注入] TextInjector
  三级回退策略：
  1. AccessibilityService 直接设置
  2. 剪贴板粘贴
  3. 模拟逐字输入
       ↓
[建议生成] SuggestionManager
  根据上下文生成 2-3 条写作建议
       ↓
[显示建议气泡] SuggestionBubbleView
  用户可点击插入
```

#### 核心模块说明

**1. 流式管道（StreamingPipeline）**

- **职责**：编排整个处理流程，确保各模块协同工作
- **技术**：Kotlin Coroutines Flow（响应式编程）
- **优势**：支持流式处理，用户体验流畅

**2. AI 引导引擎（AIGuideEngine）**

- **职责**：将用户的自然语言描述转换为结构化数据
- **技术**：Prompt Engineering + JSON Schema + JSON 修复
- **输入**：用户对人设/世界观的自然语言描述
- **输出**：结构化的 JSON 数据（人设/世界观/词库）
- **创新点**：自动提取词库，无需用户手动添加

**3. 提示词引擎（PromptEngine）- 节拍器核心**

- **职责**：根据节拍器筛选上下文，构建精准的 AI 提示词
- **技术**：模板化提示词 + 节拍器动态上下文注入
- **核心创新**：只注入当前节拍相关的人设和世界观，避免 LLM 穿帮和变笨

**节拍器上下文筛选逻辑**：

```kotlin
// 伪代码示例
fun buildPromptWithBeatContext(beat: Beat, userInput: String): String {
    // 1. 获取当前节拍关联的人设（最多 5 个）
    val relatedCharacters = beatMappingRepository
        .getCharactersForBeat(beat.id)
        .take(5)

    // 2. 获取当前节拍关联的世界观（最多 3 条）
    val relatedWorldRules = beatMappingRepository
        .getWorldRulesForBeat(beat.id)
        .take(3)

    // 3. 构建提示词（只包含相关内容）
    return """
        你是一个文本润色助手。用户正在创作小说《${project.name}》的第${beat.order}节拍。

        当前节拍：${beat.title} - ${beat.summary}
        节拍类型：${beat.type}（${getBeatTypeDescription(beat.type)}）

        相关人设（仅当前节拍）：
        ${relatedCharacters.joinToString("\n") { "- ${it.name}: ${it.summary}" }}

        相关世界观（仅当前节拍）：
        ${relatedWorldRules.joinToString("\n") { "- ${it.title}: ${it.content}" }}

        用户刚说：${userInput}

        请将以上内容润色为正式书面语，保持情节连贯，符合人物性格。
        注意：不要提及未在"相关人设"和"相关世界观"中列出的内容。
    """.trimIndent()
}
```

**示例提示词（节拍 2）**：

```
你是一个文本润色助手。用户正在创作小说《修仙传》的第 2 节拍。

当前节拍：主角林墨进入青云宗，遇到掌门
节拍类型：承（发展阶段，展开新世界）

相关人设（仅当前节拍）：
- 林墨：男，18岁，孤儿，性格坚毅，擅长剑法
- 王晓明：男，60岁，青云宗掌门，仙风道骨

相关世界观（仅当前节拍）：
- 青云宗：修仙门派，位于青云山，以剑法闻名

用户刚说：林墨走进大殿看到掌门坐在上面

请将以上内容润色为正式书面语，保持情节连贯，符合人物性格。
注意：不要提及未在"相关人设"和"相关世界观"中列出的内容。
```

**关键点**：

- ✅ 只包含 2 个人设（林墨、王晓明），不包含反派血魔
- ✅ 只包含 1 条世界观（青云宗介绍），不包含青云心法（伏笔）
- ✅ 上下文精简（约 300 tokens），LLM 理解准确
- ✅ 明确告知 LLM"不要提及未列出的内容"，避免穿帮

**4. 建议生成器（SuggestionManager）**

- **职责**：根据上下文生成写作建议
- **技术**：基于节拍类型（起/承/转/合/伏笔/高潮）的 AI 推理
- **输出**：2-3 条可直接插入的写作建议
- **创新点**：建议内容与当前情节高度相关，不是泛泛而谈
- **示例**：在"转折"节拍，AI 会建议更戏剧化的情节；在"高潮"节拍，AI 会建议更紧张的描写
